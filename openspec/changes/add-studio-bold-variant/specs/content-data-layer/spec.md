## ADDED Requirements

### Requirement: Project Schema

The data layer SHALL define a `Project` shape with required fields `title` (string), `serviceFamily` (one of the five ServiceFamily ids), `disciplines` (string array), `year`, `image`, `featured` (boolean), and `order` (number), plus optional fields `client`, `gallery` (image array), `summary`, `url`, and `accent`. Every project's `serviceFamily` MUST reference an existing ServiceFamily id, and `disciplines` MUST be drawn from that family's declared disciplines. Consumers MUST NOT depend on any field outside this schema.

#### Scenario: Valid project record
- **WHEN** a project is loaded with `title`, `serviceFamily: "visualisation"`, `disciplines: ["Architectural rendering"]`, `year`, `image`, `featured: false`, and `order`
- **THEN** the data layer returns the record with those required fields populated
- **AND** omitted optional fields (`client`, `gallery`, `summary`, `url`, `accent`) are absent or undefined rather than defaulted to placeholder strings

#### Scenario: serviceFamily must reference a known family
- **WHEN** a project declares a `serviceFamily` id that is not one of the five defined ServiceFamily ids
- **THEN** the record is rejected as invalid and is not surfaced to consumers

#### Scenario: disciplines constrained to the family
- **WHEN** a project's `disciplines` array contains a value not listed under its `serviceFamily`'s disciplines
- **THEN** the record is flagged invalid and excluded from consumer results

### Requirement: ServiceFamily Schema

The data layer SHALL define a `ServiceFamily` shape with fields `id`, `label`, `blurb`, `disciplines` (string array), `active` (boolean), and `accent` (one of the pop palette tokens). It MUST expose exactly five families in this fixed order: `visualisation`, `visual-identity`, `computational-design`, `immersive-experiences`, `digital-fabrication`. The first two MUST be `active: true` and the remaining three `active: false` ("In development"). Filters and the Services section MUST derive from this list and never hardcode family ids, labels, or disciplines.

#### Scenario: Five families in fixed order
- **WHEN** `getServices()` is called
- **THEN** it returns exactly five ServiceFamily records in the order visualisation, visual-identity, computational-design, immersive-experiences, digital-fabrication
- **AND** each record carries `id`, `label`, `blurb`, `disciplines`, `active`, and `accent`

#### Scenario: Active flags reflect the taxonomy
- **WHEN** the families are inspected
- **THEN** `visualisation` and `visual-identity` are `active: true`
- **AND** `computational-design`, `immersive-experiences`, and `digital-fabrication` are `active: false`

#### Scenario: Consumers derive from the list
- **WHEN** the WORK filter and the Services section render
- **THEN** their entries are generated by iterating `getServices()` output
- **AND** no family id, label, or discipline string is hardcoded in consumer markup

### Requirement: Source-Agnostic Access

The data layer SHALL expose `getProjects()`, `getServices()`, `getFeatured()`, and `projectsByFamily(familyId)` as the only consumer-facing read API. It MUST default to bundled placeholder JSON and MUST allow swapping the backing source to Sanity or Decap via environment configuration without any change to consumer code or to the four function signatures and return shapes.

#### Scenario: Placeholder source by default
- **WHEN** the application runs with no data-source environment configuration set
- **THEN** the four accessor functions resolve against the bundled placeholder JSON
- **AND** consumers receive Project and ServiceFamily records matching the documented schemas

#### Scenario: Swap source via environment
- **WHEN** the data-source environment variable selects Sanity or Decap
- **THEN** the accessors read from that source instead of the placeholder JSON
- **AND** consumer code calling `getProjects`/`getServices`/`getFeatured`/`projectsByFamily` is unchanged and the return shapes are identical

#### Scenario: projectsByFamily filtering
- **WHEN** `projectsByFamily("visualisation")` is called
- **THEN** it returns only projects whose `serviceFamily` equals `visualisation`
- **AND** the result is ordered by each project's `order` field

### Requirement: Featured Selection And Add-Work-Is-Data-Only

`getFeatured()` SHALL return only projects with `featured: true`, ordered by `order`, for use by the card-stack hero. Adding new work or activating a future service family SHALL be a content-only operation: creating Project records (and setting a family's `active: true`) MUST surface them through the existing accessors with no code change or rebuild of consumers.

#### Scenario: Featured filtered and ordered
- **WHEN** `getFeatured()` is called
- **THEN** it returns only projects with `featured: true`
- **AND** they are sorted ascending by `order`

#### Scenario: Adding work requires only data
- **WHEN** a new Project record is added to the active source for an already-active family
- **THEN** it appears in `getProjects()`, in `projectsByFamily()` for its family, and in `getFeatured()` when `featured: true`
- **AND** no consumer code or component is modified

#### Scenario: Activating a future family is content-only
- **WHEN** a currently in-development family is set to `active: true` and given projects in the source
- **THEN** the Services section drops its "In development" marker, the filter enables that family, and its projects render through the accessors
- **AND** no rebuild or code change beyond the data source is required
